# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A personal backup repo for Subnautica 2 save games, not an application. The only "code" is two backup scripts (one per OS) that copy save files into this repo and push them to GitHub. Everything under `Save Backups/` is generated data, not something to hand-edit.

## Commands

Run/test a backup script locally:
```
./backup-mac.sh --configure      # (re)set SAVE_PATH in backup.config
./backup-mac.sh                  # run a backup (repo <- save folder)
./backup-mac.sh --restore        # restore latest snapshot (save folder <- repo)
```
```powershell
.\backup-windows.ps1 -Configure
.\backup-windows.ps1
.\backup-windows.ps1 -Restore
```

Lint (mirrors the `shellcheck`/`psscriptanalyzer` jobs in `.github/workflows/lint.yml`):
```
shellcheck backup-mac.sh
```
```powershell
Install-Module -Name PSScriptAnalyzer -Force -Scope CurrentUser
Invoke-ScriptAnalyzer -Path backup-windows.ps1 -Severity Warning,Error -ExcludeRule PSAvoidUsingWriteHost
```
There is no unit-test suite; the closest thing is the `smoke-test-mac`/`smoke-test-windows` jobs in `lint.yml`, which run each script end-to-end against a throwaway local origin repo and a dummy save folder (first run backs up and commits, a second no-op run must not create a commit, and a deliberately corrupted script must fail loudly under `set -u` / `Set-StrictMode`). To reproduce locally, mimic those job steps.

## Architecture

- `backup-mac.sh` and `backup-windows.ps1` are independent but must stay behaviorally in sync — every change to one's backup logic should be mirrored in the other. Both follow the same flow:
  1. `--configure`/`-Configure` (or missing `backup.config`) prompts for the save folder path and writes it to `backup.config` as `SAVE_PATH=...`.
  2. Preflight checks: git installed, repo present, no rebase in progress, currently on `main`, save path exists/readable/non-empty (refuses to proceed on an empty save path so it can't wipe a real backup with nothing).
  3. `git pull --rebase origin main` to sync first; aborts the rebase and exits cleanly on failure rather than leaving the repo in a conflicted state.
  4. Deletes and recreates `Save Backups/<YYYY-MM-DD>/` (today's folder is a full mirror of the save folder, not an accumulation across runs within the same day) and copies the save folder into it.
  5. `git add` that dated folder, commit as `Backup <YYYY-MM-DD>` only if there's something staged, then `git push origin main`. A failed push leaves the commit intact locally with instructions to retry.
- `--restore`/`-Restore` is the reverse flow (import): it writes the newest snapshot in `Save Backups/` back into `SAVE_PATH`. It shares the git-installed/repo/no-rebase/on-main/best-effort-`git pull` preflight but deliberately skips the backup-only "save path exists/readable/non-empty" checks (the live save may legitimately be missing/empty when restoring onto a fresh machine). Its safety model, which must stay in sync across both scripts: pick the newest `YYYY-MM-DD` folder (lexical sort), refuse to restore from an empty snapshot, require the user to type `restore` to confirm, copy the current live save aside to a sibling `*.pre-restore-<timestamp>` folder first, then stage the copy in a sibling `*.restore-staging-<pid>` folder and swap it into place (move old aside → move staging in → roll back on failure) so an interrupted copy can never leave a half-written save folder. The `*.pre-restore-*` safety copies are intentionally left in place (not auto-deleted) as the user's undo path. Restore is not exercised by the smoke tests.
- `backup.config` holds the real `SAVE_PATH` and is gitignored (machine-specific); `backup.config.example` is the committed template.
- `Save Backups/<date>/` folders are the actual save snapshots, one per day the backup was run, keyed by calendar date (not per-run).
- `savecode.txt` is a hand-maintained, committed note holding the game's save/friend code — not generated by the scripts and unrelated to the backup flow.

## CI and automation

Four GitHub Actions workflows, all keyed off `main`:

- `lint.yml` ("Lint backup scripts") — runs on pushes to `main` that touch either backup script or any of the workflow files, and on *every* PR unconditionally (deliberately no path filter on the `pull_request` trigger — `auto-merge.yml` gates on this workflow completing for the PR's head SHA, so a path filter that skipped a PR would make auto-merge wait on a run that never happens). Jobs: `shellcheck` (backup-mac.sh), `psscriptanalyzer` (backup-windows.ps1), `smoke-test-mac`, `smoke-test-windows` (the end-to-end runs described above), and `notify-on-failure`. On failure `notify-on-failure` either comments `@claude` on the failing PR, or (for push events) opens/updates a `ci-failure`-labeled tracking issue tagging `@claude`.
- `codeql.yml` ("CodeQL Advanced") — CodeQL scanning of the GitHub Actions workflows themselves (`language: actions`) on push/PR to `main` (no path filter either, for the same auto-merge reason) and on a weekly schedule. On scheduled runs, `notify-on-alert` opens/updates a `codeql-alert`-labeled issue tagging `@claude` when there are open alerts.
- `claude.yml` ("Claude Code") — runs `anthropics/claude-code-action` when an issue or comment mentions `@claude`, and opens a PR for any branch Claude pushes. Commits as `subnautica2-backup-bot` (not the triggering user) via `ssh_signing_key`/`bot_id`/`bot_name`, so they show as GitHub-verified; `claude_args` explicitly forbids the `Co-authored-by` trailer the action would otherwise add for the triggering user, since that has no dedicated on/off input and previously downgraded commits from "Verified" to "Partially verified".
- `auto-merge.yml` ("Auto-merge Claude PRs") — triggers on completion of either `lint.yml` or `codeql.yml`; each firing re-checks *both* workflows' latest run for that commit SHA (not just the one that just completed) and only squash-merges + deletes the branch once every required workflow has succeeded. Keep the `workflows:` list here and the `required` array in its merge step in sync if a new required check is added.

Notes for editing workflows: the automation deliberately uses a PAT (`secrets.BOT_GH_PAT`, belonging to the `subnautica2-backup-bot` account) instead of the default `GITHUB_TOKEN` or a personal PAT, wherever one workflow needs to trigger another, keep `@claude`-mention and `Fixes #N` auto-close linkages working, or attribute commits/merges/comments to the bot rather than a human. Don't swap it back to `github.token`, and don't swap it back to a personal PAT either. This creates a self-healing loop (CI failure → `@claude` → fix pushed → CI → auto-merge), so touching these files can have knock-on effects; the inline comments in each workflow explain the non-obvious token and parsing choices.
